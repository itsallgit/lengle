# Release Plan — v1.16: Branch-First Release Workflow

## Overview

| Field   | Value                                           |
|---------|------------------------------------------------|
| Release | v1.16                                          |
| Branch  | release/v1.16                                  |
| Summary | Overhaul the agentic release workflow so release branches are created before planning begins, eliminating the `plans/draft.md` staging concept, introducing incremental agent-prefixed commits throughout the lifecycle, and adding guided agent handovers with user checkpoints. |

---

## Problem Statement

The current workflow writes `plans/draft.md` on `main` and only creates a release branch after design is complete. This causes three problems:

1. **Stranded drafts** — If a release closes without deleting its draft, or if the feature is deferred, `draft.md` accumulates on `main` with no release context.
2. **Late branching** — The release branch appears only after planning and design are done, so the plan's evolution is not captured in branch history.
3. **Infrequent commits** — Work is committed in large batches (typically after deploy) rather than incrementally at each agent handoff.

---

## Acceptance Criteria

1. **No more `draft.md`** — The concept of `plans/draft.md` is eliminated from all agent instructions, skills, specs, and documentation. Plans are always written directly as `plans/vX.Y.0-release.md` (or `plans/vX.Y.Z-hotfix.md`) on a release branch.

2. **Plan naming convention** — Plan files use version-first naming: `plans/vX.Y.0-release.md` for releases, `plans/vX.Y.Z-hotfix.md` for hotfixes. This replaces the old `plans/release-vX.Y.md` and `plans/draft.md` conventions.

3. **Hotfix plan files** — Hotfixes get their own plan files (`plans/vX.Y.Z-hotfix.md`) on the hotfix branch, following the same plan → design → build lifecycle as releases.

4. **Branch created before planning** — When a user describes a new change and no active release exists, the orchestrator routes to Release Agent to create the branch first, then routes to Plan Agent to write the plan on that branch.

5. **Active-release awareness** — When a user describes a new change and an active release branch exists, the orchestrator informs the user and asks whether the change should be added to the active release or deferred.

6. **Agent-prefixed commits** — All branch commits use the agent name as prefix: `plan(vX.Y):`, `design(vX.Y):`, `build(vX.Y):`, `release(vX.Y):`. The old `wip(vX.Y):` prefix is eliminated. Production backups on `main` use `backup:` prefix.

7. **Every agent commits before handoff** — Each agent commits its own work before completing. No handoff with uncommitted changes.

8. **Commit trail is a readable narrative** — A typical release branch history reads:
   ```
   release(vX.Y): create release branch
   plan(vX.Y): add release plan
   design(vX.Y): add technical design
   build(vX.Y): phase 1 — [description]
   build(vX.Y): phase 2 — [description]
   release(vX.Y): deploy to non-prod
   release(vX.Y): snapshot game data
   vX.Y: [release summary]
   ```

9. **Guided handovers with user checkpoints** — When one agent finishes, the orchestrator presents a summary block showing: what the agent completed, the commit it made, and what the recommended next step is. The user acknowledges before the next agent starts. This gives the user an opportunity to review progress and provide feedback at every stage.

10. **Pause after design** — The orchestrator always pauses after Design Agent completes, giving the user a chance to review the technical plan before Build Agent begins implementation.

11. **Smart close-release checks** — Close-release always requires a What's New entry. Code checks (`typecheck` + `lint`) are only required when code files under `app/src/` were modified on the branch.

12. **Game data snapshots on the release branch** — Before closing a release, the latest production game data is pulled and committed on the release branch (`release(vX.Y): snapshot game data`). This data is preserved in the squash-merge and tag, eliminating standalone backup commits on `main` during the release lifecycle.

13. **Production backup on main** — The Production Agent still snapshots game data before deploying to production (after the squash merge). This is the one case where a backup commit goes on `main`, using the `backup: game data YYYYMMDD-HHMMSS` prefix.

14. **What's New for non-user-facing releases** — Agentic solution updates and other non-user-facing changes use the 🤖 emoji in their What's New entry to distinguish them from feature releases.

15. **Stale draft removed** — The existing `plans/draft.md` (Puzzle Words feature, already shipped) is deleted.

16. **All documentation consistent** — Every file that references the old workflow is updated. No contradictory instructions remain.

## Scope Notes

- **Historical release plans are not modified.** Files like `plans/release-v1.13.md` are historical records and retain their original content even though they describe the old workflow and use the old naming convention.
- **Build Agent gains git commit capability.** Its constraint "NEVER run git commit" is relaxed to allow `build(vX.Y):` commits after each phase.
- **Production Agent gains backup commit prefix change.** Its backup commit format changes from `chore: backup game data` to `backup: game data YYYYMMDD-HHMMSS`.

## Files to Change

### 1. `.github/agents/orchestrator.agent.md`

**What changes:**

- **Handoff prompt for Plan Agent** — Remove reference to `plans/draft.md`. Update to: "Plan this request on the active release branch."
- **State detection** — Replace the `draft.md` existence check with:
  1. Run `git branch --show-current`
  2. Check for active release/hotfix branches: `git branch -a --list 'release/*' --list 'hotfix/*'`
  3. If on a release/hotfix branch, check that the matching plan file exists
- **Routing rules** — Replace the three `main` + `draft.md` rules with:
  - On `main` with no active release branch and user describes a change → Route to Release Agent to create branch, then route to Plan Agent to write the plan on the branch.
  - On `main` with an active release branch and user describes a change → Ask user whether to add to the existing release or defer. If adding, switch to the release branch and route to Plan Agent to update the plan.
  - On `release/vX.Y` or `hotfix/vX.Y.Z` with a plan but no Technical Implementation section → Design Agent.
  - On `release/vX.Y` or `hotfix/vX.Y.Z` with a fully designed plan → Build Agent (after user acknowledges the design).
  - On `release/vX.Y` or `hotfix/vX.Y.Z` for deploy/status/close requests → Release Agent.
  - On any branch for production operations → Production Agent.
  - "Where were we?" on `main` with a remote release/hotfix branch → Inform user and offer to switch.
- **Guided handover protocol** — Add a new section describing the handover behaviour:
  - When an agent completes, present a summary block to the user showing: what was done, the commit reference, and the recommended next step.
  - Wait for user acknowledgement before routing to the next agent.
  - Always pause after Design Agent completes for technical plan review.
- **Chaining rules** — Simplify to: "After Release Agent creates a branch, recommend Plan Agent as the next step." Remove any auto-chain rules. All transitions go through user acknowledgement via the handover protocol.

### 2. `.github/agents/release-agent.agent.md`

## Decisions & Scope

| Decision | Choice | Rationale |
|---|---|---|
| Plan naming convention | `vX.Y.0-release.md` / `vX.Y.Z-hotfix.md` | Version-first naming makes the version immediately visible and distinguishes releases from hotfixes |
| Hotfix plan files | Yes, always | Hotfixes follow the same plan → design → build lifecycle; having a plan file keeps the process consistent |
| Historical release plans | Leave unchanged (old naming) | They are historical records; renaming them would be revisionist |
| Backup commit format | `backup:` prefix (no agent name) | Backups are data snapshots, not agent-specific work; `backup:` is descriptive and concise |
| Pre-deploy production backup | On `main` with `backup:` prefix | This is the one exception; it captures data state between squash-merge and production deploy |
| Game data snapshot at close | On release branch with `release(vX.Y):` prefix | Preserved in the squash-merge; eliminates standalone backup commits during the release lifecycle |
| Agent handover model | Guided with user checkpoints | User sees a completion summary and acknowledges before the next agent starts |
| Auto-chaining | Eliminated | All transitions go through user acknowledgement; orchestrator recommends but never auto-chains |
| Post-design pause | Always | Technical plan is the most consequential output; user reviews before implementation begins |
| Build Agent git access | Allow `git commit` and `git add` only | Minimal expansion; push/merge/checkout remain Release Agent's domain |
| Close-release code checks | Conditional on app code changes | No point running typecheck/lint if only docs/config changed |
| What's New for non-user-facing | 🤖 emoji | Distinguishes agentic/internal releases from user-facing features |
| Deferred changes | User told to wait | Only one release branch at a time per git-standards; no queueing mechanism needed |
