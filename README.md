# Lengle

A private daily word puzzle game for three players. Each player sets a secret 5-letter word for the others to guess. Feedback is numeric rather than colour-based — every guess scores +0 (correct position), +1 (correct letter, wrong position), or +3 (not in word) per letter. The player who solves both daily puzzles in the fewest total guesses wins the day.

Built as a fully static React app hosted on AWS S3 + CloudFront. No backend, no database, no auth — all game state is stored as JSON files in S3.

## Part 1 — AWS CLI setup

You will need the AWS CLI installed and configured before deploying any infrastructure. Follow these steps on your new machine.

### Step 1 — Install the AWS CLI

**macOS (recommended — Homebrew):**
```bash
brew install awscli
```

**macOS (official installer):**

Download and run the `.pkg` installer from:
https://docs.aws.amazon.com/cli/latest/userguide/install-cliv2-mac.html

**Windows:**

Download and run the `.msi` installer from:
https://docs.aws.amazon.com/cli/latest/userguide/install-cliv2-windows.html

**Verify the installation:**
```bash
aws --version
# Expected output: aws-cli/2.x.x Python/3.x.x ...
```

---

### Step 2 — Create a dedicated IAM user in AWS

Never use your root account for deployments. Create a dedicated IAM user for Lengle.

1. Sign in to the AWS Console at https://console.aws.amazon.com
2. Navigate to **IAM** → **Users** → **Create user**
3. Set the username to `lengle-deploy`
4. On the permissions screen, select **Attach policies directly**
5. Search for and attach **AdministratorAccess**
6. Complete the user creation
7. Open the newly created user → **Security credentials** tab
8. Under **Access keys**, click **Create access key**
9. Select **Command Line Interface (CLI)** as the use case
10. Click through and **download the `.csv` file** or copy both keys somewhere safe — you will not be able to see the secret key again

> You will have two values: an **Access Key ID** (starts with `AKIA...`) and a **Secret Access Key**. Keep these private.

---

### Step 3 — Configure a named AWS profile

Run the following and enter your keys when prompted:

```bash
aws configure --profile lengle
```

You will be asked for four values:

| Prompt | Value |
|---|---|
| AWS Access Key ID | Your Access Key ID from Step 2 |
| AWS Secret Access Key | Your Secret Access Key from Step 2 |
| Default region name | `ap-southeast-2` |
| Default output format | `json` |

### Step 4 — Verify the profile works

```bash
aws sts get-caller-identity --profile lengle
```

Expected output:
```json

    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/lengle-deploy"

```

If you see your Account ID and user ARN, your credentials are working correctly.

### Step 5 — Set the profile as your active profile

Export the profile in your shell so all subsequent AWS and CDK commands use it automatically:

Verify it is active:

### Step 15 — Session 5a: Lobby components

Start a new chat. Attach both specs. Send:

```
Read specs/spec-implementation.md and specs/spec-game-design.md, then build
3 files only — the Lobby components. Do not build any Puzzle components yet.

Files to create:
- app/src/components/Lobby/PlayerStatusList.tsx
- app/src/components/Lobby/WordSetForm.tsx
- app/src/components/Lobby/Lobby.tsx

Architecture decisions already resolved — do not re-derive them:

POLLING:
- Use a single useEffect with setInterval in Lobby.tsx for polling (do not use
  useS3Poll for the lobby — manage the interval directly for full control)
- Poll status.json every 30 seconds only while isUnlocked is false
- On mount, do an immediate fetch of status.json before the interval starts
- Store a cancelled flag (ref) to prevent state updates after unmount
- Stop polling as soon as status.unlocked === true

STATE MACHINE:
- Lobby has exactly three states: STATE_A (player has not set today's word),
  STATE_B (player has set word, others pending), STATE_C (all words set, unlocked)
- Derive the state from: (a) whether the player's own word file exists for today,
  and (b) status.json. Do not use optimistic local flags.
- On mount, fetch both status.json AND data/words/{date}/{playerId}.json in parallel
  to determine the initial state correctly without any loading flicker

WORD SUBMISSION:
- On submit: write data/words/{date}/{playerId}.json first, then read current
  status.json (may be null if first player), merge in the new player's key,
  recompute unlocked (true when all 3 players have words_set === true), write back
- Last-write-wins is safe for status.json — each player writes exactly once

USED WORDS VALIDATION:
- To check uniqueness: call listS3Keys('data/words/') to get all past word file keys,
  filter to keys where the date portion is before today's active puzzle date,
  fetch each via readViaCloudFront in parallel, collect all words into a Set<string>
- Load this set on mount. If not loaded yet when player submits, skip the uniqueness
  check (acceptable edge case for a family game)

TOMORROW'S WORD:
- On mount, also fetch data/words/{tomorrowDate}/{playerId}.json to check if it
  is already set. Show collapsible section. Once set, show read-only confirmation.
- Tomorrow's date = getActivePuzzleDate() incremented by one calendar day

Do not build any Puzzle components. Only these 3 Lobby files.
```

## Troubleshooting

**CORS errors in browser console**
The S3 bucket CORS policy may not include your S3 website URL. Re-run `cdk deploy` after updating `AllowedOrigins` in the stack (Step 24).

**Blank page or wrong content served**
Confirm S3 static website hosting is enabled on the bucket and that the index document is set to `index.html`. Check the bucket policy allows public `s3:GetObject`. Verify `VITE_S3_WEBSITE_URL` in `app/.env.local` matches the `WebsiteUrl` output from CDK.

**Cognito credential errors**
Verify `VITE_COGNITO_IDENTITY_POOL_ID` in `app/.env.local` matches the `IdentityPoolId` output from CDK. Rebuild and redeploy the app after changing `.env.local`.

**Word History is blank or returns 403**
The Cognito unauthenticated IAM role requires `s3:ListBucket` permission on the bucket to run `ListObjectsV2`. If this permission is missing, Word History and results finalisation will silently fail. Check `infra/lib/lengle-stack.ts` and confirm the unauthenticated role includes `s3:ListBucket` with an `s3:prefix` condition scoped to `data/`.

**Stale data (changes not appearing)**
The CloudFront cache for `/data/*` paths has a TTL of 0 (disabled). If you are seeing stale data, do a hard refresh (`Cmd+Shift+R` / `Ctrl+Shift+R`). If the problem persists, create a manual CloudFront invalidation for `/data/*`.

**`cdk deploy` fails with "Unable to resolve AWS account"**
Your AWS profile is not active. Run `export AWS_PROFILE=lengle` and try again.

**`cdk bootstrap` required message**
Run `cdk bootstrap aws://YOUR_ACCOUNT_ID/ap-southeast-2` before deploying (Step 9).

**Two players submitted the same word for tomorrow**
This is a known limitation. The uniqueness check for puzzle words only validates against past days — the app cannot see other players' future words by design (word security). If this happens, one player will need to change their word. Ask that player to open the Lobby, expand "Set Tomorrow's Word", and submit a different word. Note: the first submission is locked, so this requires manually deleting the conflicting word file from S3:
```bash
aws s3 rm s3://$BUCKET_NAME/data/words/YYYY-MM-DD/player_X.json
```
Then the player can re-submit via the app.

## Recovering data

S3 versioning is enabled on the data bucket. To restore a previous version of a game data file:

```bash
# List versions of a file
aws s3api list-object-versions \
  --bucket $BUCKET_NAME \
  --prefix data/days/2026-04-01/guesses-player_1.json

# Restore a specific version
aws s3api copy-object \
  --bucket $BUCKET_NAME \
  --copy-source "$BUCKET_NAME/data/days/2026-04-01/guesses-player_1.json?versionId=<VERSION_ID>" \
  --key data/days/2026-04-01/guesses-player_1.json
```

## Adding CloudFront and a custom domain (post-MVP)

CloudFront adds HTTPS, global caching, and enables custom domains. To enable it:

1. Edit `infra/cdk.json` to enable CloudFront:
   ```json
   {
     "context": {
       "cloudfront": {
         "enabled": true,
         "customDomain": {
           "enabled": false,
           "domainName": "",
           "hostedZoneId": ""
         }
       }
     }
   }
   ```
2. Run `cdk deploy`
3. Update `VITE_S3_WEBSITE_URL` in `app/.env.local` to the new CloudFront HTTPS domain from CDK outputs
4. Rebuild and redeploy the app: `bash scripts/deploy.sh`

To also add a custom domain:

1. Register a domain (AWS Route 53 or any registrar)
2. Create a Route 53 Hosted Zone for the domain
3. Note the Hosted Zone ID from the Route 53 console
4. Set `customDomain.enabled: true` in `cdk.json`, add `domainName` and `hostedZoneId`
5. Run `cdk deploy`
6. If using an external registrar, update your domain's nameservers to the Route 53 nameservers

DNS propagation can take up to 48 hours, though it is usually much faster.
